# Projeto — Sistema Modular de Controle de Licenças/COREN com Notificações WhatsApp

## 1. Visão geral

O projeto será uma **plataforma web modular**, construída do zero, com foco em controlar licenças/documentos profissionais, especialmente **COREN**, acompanhar vencimentos, permitir envio e validação de comprovantes e disparar notificações automáticas para profissionais e responsáveis.

A referência inicial veio do arquivo **“Projeto Whatsapp.pdf”**, que apresenta uma solução de atendimento via WhatsApp com agentes, supervisores, dashboard, relatórios, status, menus e mensagens automáticas. Porém, o projeto atual **não será um clone do Pysix/call center**.

O PDF será usado apenas como inspiração para:

* dashboard operacional;
* status;
* relatórios;
* mensagens automáticas;
* divisão por responsáveis/times;
* visão de acompanhamento gerencial.

O contexto inicial também indica duas frentes possíveis:

1. Um sistema inspirado no modelo do PDF, com possível relação com metas/pagamento.
2. Um sistema para controle de vencimento de COREN, com alertas para RT de enfermagem e profissionais.

A decisão consolidada é focar na **V1 operacional apresentável** do sistema de controle de licenças/COREN, deixando atendimento estilo call center, metas e pagamento como módulos opcionais futuros, somente se forem confirmados e pagos.

---

# 2. Objetivo principal

Criar um sistema enxuto, real e apresentável para:

> Controlar profissionais, licenças e vencimentos, permitir upload de documentos, validação por responsáveis e envio automático de notificações via WhatsApp oficial da Meta, com histórico persistido no banco e relatórios operacionais completos desde a primeira versão.

Este sistema não será um protótipo descartável.

A ideia central é:

> **Escopo enxuto, arquitetura definitiva.**

Ou seja: a V1 não precisa conter todos os módulos imagináveis, mas tudo que for construído desde o dia 0 deve ser feito no formato correto para continuar sendo usado no produto final.

---

# 3. Princípios do projeto

## 3.1 V1 operacional apresentável

A partir deste consolidado, a palavra “MVP” deve ser entendida com cuidado.

O objetivo não é fazer uma versão improvisada para depois refatorar.

O objetivo é construir uma **V1 operacional apresentável**, com escopo controlado, mas com arquitetura, banco, permissões, notificações, relatórios, auditoria e fluxos já pensados como base definitiva.

A V1 precisa nascer com:

* baixo custo de infraestrutura;
* código modular;
* banco persistente;
* autenticação para superiores;
* dashboard;
* notificações WhatsApp oficiais;
* upload de documentos;
* validação;
* histórico;
* auditoria;
* relatórios operacionais completos;
* FAQ/suporte simples via `wa.me`;
* possibilidade real de expansão sem refazer o miolo.

## 3.2 Não fazer cada coisa duas vezes

Este projeto será desenvolvido com um pipeline pessoal agentic. Cada tarefa vai consumir tokens, tempo e energia.

Portanto, as tarefas devem evitar o padrão:

```txt
faz simples agora
↓
descobre que não sustenta
↓
refaz depois
```

O padrão correto deve ser:

```txt
modela certo desde o início
↓
implementa apenas o escopo necessário
↓
mantém pontos de expansão limpos
↓
adiciona módulos futuros sem quebrar o núcleo
```

Isso vale especialmente para:

* modelagem do banco;
* autenticação/roles;
* profissionais;
* licenças;
* documentos;
* upload tokenizado;
* notificações;
* logs;
* webhooks;
* auditoria;
* relatórios.

## 3.3 WhatsApp oficial, mas sem intermediário

A decisão consolidada é:

```txt
Meta WhatsApp Cloud API: sim
Twilio/Zenvia/provedor intermediário: não na V1
Automação não oficial via WhatsApp Web: não
```

A Meta será o custo inevitável para envio automático real via WhatsApp.

O restante será **BIH — built in-house**.

Fluxo consolidado:

```txt
Sistema
↓
NotificationJob no banco
↓
Worker/cron diário
↓
Monta template
↓
Chama Meta WhatsApp Cloud API diretamente
↓
Recebe webhook da Meta
↓
Salva status no PostgreSQL
```

Isso deve ser considerado arquitetura definitiva do módulo de notificações, não gambiarra de MVP.

## 3.4 Modularidade sem microserviços

O sistema será um **monólito modular**.

Ou seja:

* um backend;
* um banco;
* uma API;
* deploy simples;
* módulos internos bem separados.

Nada de microserviços no começo.

Nada de dividir infraestrutura antes da necessidade real.

A modularidade deve existir na organização do código, nos limites de domínio e na clareza dos módulos, não na fragmentação prematura da infra.

# 4. Stack consolidada

## 4.1 Frontend

```txt
React
TypeScript
Tailwind CSS
Vite
```

A decisão atual é seguir com **React + TypeScript + Tailwind**, preferencialmente com Vite.

Next.js fica fora da recomendação principal neste momento, porque o projeto não exige SSR, rotas server-side ou fullstack acoplado para a V1.

A interface precisa ser:

* web;
* responsiva;
* apresentável;
* boa para dashboard;
* boa para tabelas;
* boa para formulários;
* simples de manter.

Não começar com React Native/Expo.

A ideia de app mobile pode voltar no futuro, mas não entra na V1 porque adicionaria custo, complexidade e manutenção.

## 4.2 Backend

```txt
Node.js
Express
TypeScript
Prisma
PostgreSQL
```

O backend deve ser simples, modular e previsível.

Express é suficiente para a V1, desde que o projeto seja organizado por módulos e camadas.

## 4.3 Banco

```txt
PostgreSQL
Prisma ORM
```

PostgreSQL é a escolha principal.

Prisma será usado como ORM para modelagem, migrations e acesso tipado ao banco.

A modelagem precisa nascer bem feita desde o início, porque o banco será o coração do sistema e o principal ponto onde rework deve ser evitado.

## 4.4 Storage

Para documentos/licenças:

```txt
Supabase Storage
Cloudflare R2
ou S3-compatible equivalente
```

Não salvar arquivo diretamente no banco.

O banco salva apenas:

* metadados;
* chave do arquivo;
* URL interna/privada;
* status;
* vínculo com profissional/licença;
* dados de validação.

Os arquivos devem estar em storage externo, preferencialmente privado, com controle de acesso por backend.

## 4.5 Notificações

```txt
WhatsApp:
Meta WhatsApp Cloud API direta

E-mail:
Opcional/fallback futuro, não obrigatório na V1

Dashboard:
Notificação interna obrigatória
```

WhatsApp será obrigatório porque o público provavelmente não verificará e-mail de forma consistente.

E-mail pode ficar previsto como canal futuro ou fallback, mas não deve ser premissa da V1 se isso encarecer ou dispersar o foco.

## 4.6 Jobs

Na V1:

```txt
Cron diário
Tabela NotificationJob no PostgreSQL
Worker simples no próprio backend ou processo separado leve
```

Sem Redis/BullMQ no início.

A modelagem, porém, deve permitir que no futuro um worker mais robusto seja acoplado sem reescrever o domínio de notificações.

# 6. Escopo funcional da V1 operacional apresentável

## 6.1 Autenticação

O sistema terá login apenas para usuários administrativos/superiores.

Perfis iniciais:

```txt
ADMIN
RT
SUPERVISOR
```

O profissional **não terá login completo na V1**.

Para reduzir atrito, a profissional acessará por **link seguro/token** para:

* visualizar pendência específica;
* enviar documento;
* reenviar documento se recusado;
* acessar ajuda/FAQ;
* acionar suporte via `wa.me`.

Decisão importante:

```txt
User ≠ Professional
```

* `User` = quem loga no sistema.
* `Professional` = pessoa monitorada pelo sistema.

Para evitar rework futuro, pode existir um campo opcional:

```txt
Professional.userId nullable
```

Na V1 ele provavelmente ficará nulo.

Se no futuro existir portal da profissional com login, será possível vincular sem refazer o modelo.

## 6.2 Gestão de usuários

Funcionalidades:

* criar usuário;
* editar usuário;
* ativar/desativar usuário;
* definir role;
* associar RT/supervisor a setores/unidades;
* resetar senha;
* controlar acesso por organização/unidade/setor;
* impedir exclusão destrutiva se houver histórico associado.

Perfis:

```txt
ADMIN:
acesso amplo à organização

RT:
acesso às profissionais sob sua responsabilidade

SUPERVISOR:
acesso aos setores/unidades associados
```

## 6.3 Gestão de organizações, unidades e setores

Mesmo que o primeiro cliente seja uma única instituição, a estrutura deve nascer com:

```txt
Organization
Unit
Sector
```

Isso evita retrabalho se existirem:

* múltiplas unidades;
* múltiplos setores;
* múltiplas RTs;
* diferentes responsáveis por área;
* futura replicação do sistema para outro cliente.

Não é necessário criar um SaaS multi-tenant completo na V1, mas as tabelas centrais devem ter `organizationId` desde o começo.

## 6.4 Gestão de profissionais

Campos sugeridos:

```txt
Nome
CPF, se necessário
E-mail
Telefone/WhatsApp
Cargo
Setor
Unidade
RT responsável
Status ativo/inativo
Observações
```

Funcionalidades:

* cadastrar profissional;
* editar profissional;
* listar profissionais;
* filtrar por status;
* filtrar por RT;
* filtrar por setor;
* filtrar por unidade;
* visualizar histórico da profissional;
* visualizar licenças associadas;
* visualizar documentos enviados;
* visualizar notificações enviadas;
* desativar profissional sem apagar histórico.

Importação CSV/Excel pode ser tratada como módulo opcional posterior, mas o modelo precisa estar pronto para receber dados em massa sem alterar estrutura.

## 6.5 Gestão de licenças/documentos

A entidade deve ser genérica, não presa apenas a COREN.

Tipos possíveis:

```txt
COREN
Certificado
Treinamento obrigatório
Documento profissional
Outro
```

Campos da licença:

```txt
Tipo da licença
Número
UF/emissor
Data de emissão
Data de vencimento
Profissional associado
Status calculado
Última validação
Validado por
Observações
```

Decisão importante:

```txt
Professional
  has many Licenses

License
  has many Documents
```

Não salvar apenas “último arquivo” na licença.

A licença deve ter histórico de documentos, porque a profissional pode:

* enviar documento errado;
* reenviar documento;
* ter documento recusado;
* enviar atualização;
* ter nova validação.

## 6.6 Upload de documentos por link mágico

Fluxo:

```txt
Profissional recebe link
↓
Abre tela simplificada
↓
Confere dados básicos
↓
Faz upload do documento/comprovante
↓
Sistema registra envio
↓
Documento vira UPLOADED
↓
Licença pode virar PENDING_VALIDATION
↓
RT/Admin valida ou recusa
```

Funcionalidades:

* upload de PDF/imagem;
* limite de tamanho;
* validação de tipo de arquivo;
* status do documento;
* histórico de uploads;
* recusa com motivo;
* aprovação com data e usuário responsável;
* token com expiração;
* token armazenado como hash;
* token invalidado após uso, quando fizer sentido;
* possibilidade de gerar novo token para reenvio.

## 6.7 Cálculo automático de status

O sistema calcula status com base na data de vencimento, no status do documento e na validação.

Exemplo:

```txt
Vencimento futuro acima do limite configurado:
REGULAR

Vence dentro de X dias:
EXPIRING

Vencimento anterior à data atual:
EXPIRED

Não há documento válido associado:
PENDING_DOCUMENT

Documento enviado e não validado:
PENDING_VALIDATION

Profissional desativada ou licença não aplicável:
INACTIVE
```

Os prazos precisam ser configuráveis.

Exemplo:

```txt
Avisar 60 dias antes
Avisar 30 dias antes
Avisar 15 dias antes
Avisar 7 dias antes
Avisar no dia do vencimento
Repetir após vencido a cada X dias
```

Essas regras não devem ficar hardcoded dentro do cron.

Elas devem ser representadas por `NotificationRule`.

# 9. Notificações WhatsApp

Esse é o núcleo mais sensível do projeto.

A V1 deve usar WhatsApp oficial via Meta Cloud API, sem Twilio/Zenvia/intermediários.

## 9.1 Fluxo consolidado

```txt

↓

↓

↓

↓
Worker busca jobs pendentes
↓
Monta payload com template aprovado
↓
Chama Meta WhatsApp Cloud API
↓
Recebe resposta da Meta
↓
Salva providerMessageId
↓
Webhook recebe status
↓
Atualiza NotificationLog
↓
Atualiza NotificationJob
↓
Atualiza dashboard/relatórios
```

# 19. Riscos principais

## 19.1 WhatsApp

Riscos:

* templates recusados;
* mudança de preço;
* falha de entrega;
* número não habilitado;
* telefone inválido;
* bloqueio por má qualidade de mensagens;
* necessidade de configuração no Business Manager;
* webhook mal configurado;
* erro não rastreado.

Mitigação:

* templates simples e utilitários;
* logs detalhados;
* dashboard de falhas;
* relatórios de notificação;
* não enviar marketing;
* não disparar spam;
* manter `NotificationJob` e `NotificationLog`;
* guardar `providerMessageId`.

## 19.2 LGPD e dados sensíveis

O sistema pode armazenar dados pessoais e documentos.

Mitigação:

* controle de acesso;
* storage privado;
* links temporários;
* token com hash;
* auditoria;
* não deletar histórico crítico;
* limitar exposição por role;
* evitar exibir dados sensíveis desnecessários;
* desativar em vez de apagar registros com histórico.

## 19.3 Escopo crescer

O PDF pode criar expectativa de call center completo.

Mitigação:

* deixar explícito que Pysix é referência, não escopo;
* documentar não objetivos;
* tratar call center como módulo futuro;
* fechar V1 por escrito.

## 19.4 Custo de infraestrutura

Mitigação:

* monólito modular;
* cron simples;
* Postgres serverless;
* storage barato;
* sem Redis no começo;
* sem app mobile no começo;
* sem Twilio/Zenvia no começo;
* sem call center.

## 19.5 Rework no pipeline agentic

Risco:

* tarefas pequenas demais sem visão final;
* agentes criando tabelas simplificadas demais;
* features implementadas direto no controller sem domínio;
* notificações hardcoded;
* relatórios sem histórico;
* permissões improvisadas.

Mitigação:

* este consolidado é fonte da verdade;
* cada tarefa deve respeitar o modelo final;
* cada módulo deve nascer com seu domínio correto;
* evitar atalhos que invalidem o banco;
* manter “escopo enxuto, arquitetura definitiva”.

# 21. Diretrizes para tarefas agentic

Cada tarefa gerada para o pipeline deve seguir estas regras:

## 21.1 Não simplificar contra o domínio

Errado:

```txt
Criar tabela funcionarios com nome, telefone e vencimento_coren.
```

Certo:

```txt
Criar módulo Professional vinculado a Organization, Unit, Sector e responsibleRtId, mantendo Professional separado de User e preparado para múltiplas licenças.
```

## 21.2 Não hardcodar notificações

Errado:

```txt
Cron busca licenças vencidas e chama Meta direto.
```

Certo:

```txt
Cron cria NotificationJobs com base em NotificationRules; worker processa jobs usando NotificationTemplate e MetaWhatsAppProvider.
```

## 21.3 Não misturar status de licença com status de documento

Errado:

```txt
status = aguardando validação no registro da licença sem distinguir documento.
```

Certo:

```txt
License tem LicenseStatus; Document tem DocumentStatus; ambos se relacionam no fluxo.
```

## 21.4 Relatórios dependem de histórico

Toda tarefa que altera dados importantes deve também considerar:

* audit log;
* timestamps;
* usuário responsável;
* metadados;
* status anterior/novo quando relevante.

## 21.5 Interface simples, mas definitiva

A UI pode ser simples, mas não deve parecer rascunho.

A V1 será apresentada.

Então telas principais precisam ter:

* layout limpo;
* tabelas usáveis;
* filtros;
* estados vazios;
* estados de erro;
* feedback visual;
* ações claras.

# 22. Resumo executivo

O projeto será uma plataforma web modular para controle de licenças profissionais, começando pelo COREN. O sistema terá React, TypeScript e Tailwind no frontend, Node.js, Express, Prisma e PostgreSQL no backend.

A V1 operacional apresentável terá login para superiores, cadastro de organização/unidade/setor, cadastro de profissionais, cadastro de licenças, cálculo de vencimentos, upload de documentos por link mágico, validação por RT/admin, dashboard, relatórios operacionais completos, auditoria e notificações automáticas via WhatsApp usando diretamente a Meta WhatsApp Cloud API.

A integração com WhatsApp será oficial e paga via Meta, mas sem Twilio ou intermediários na V1. Toda a camada de jobs, templates, logs, webhooks, auditoria e persistência será construída internamente.

O Pysix será usado apenas como inspiração para dashboard, status, relatórios e mensagens automáticas. A V1 não terá call center, fila de atendimento, agentes online, bot, PABX, chat interno, metas/pagamento ou app mobile.

A visão final é um sistema magro, extensível e de baixo custo, começando pelo núcleo de conformidade/licenças, mas já construído com miolo definitivo para evitar rework no pipeline agentic.

A frase-guia do projeto é:

> **Escopo enxuto, arquitetura definitiva.**
